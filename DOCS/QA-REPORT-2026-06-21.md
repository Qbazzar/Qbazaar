## 2) الباجات المكتشفة (الجذر · الإصلاح · التحقق)

### 2.1 — `send-otp` يرجّع 500
- **الجذر:** `OtpNotification` كان يُرسَل **متزامناً**؛ عند ارتداد بريد OTP (عنوان غير قابل للتسليم → 550) يتصاعد الخطأ إلى استجابة 500.
- **الإصلاح:** جعل `OtpNotification` يطبّق `ShouldQueue` ليُعالَج عبر Horizon — الارتداد صار **مهمّة فاشلة** لا طلباً فاشلاً. يحاكي إشعارات Welcome/SecurityAlert.
- **التحقق:** اختبار انحدار + جسّ حي → `send-otp` يرجّع **202**.
- **الكوميت:** `981473c`

### 2.2 — رفع الصور `r is not iterable`
- **الجذر:** نقطة الباك ترجّع `{ success, data: { images: Media[] } }`، لكن عميل الفرونت قرأ `data.data` (الكائن الغلاف) ونوّعه كمصفوفة. الـdropzone نفّذ `[...images, ...uploaded]` على كائن → `TypeError: ... is not iterable` يحجب كل رفع صورة.
- **الإصلاح:** قراءة `data.data.images` (مع `?? []`) + اختبار Vitest للعقد.
- **الكوميت:** `dc5ea22`

### 2.3 — تعديل المسودة 404
- **الجذر:** راوت `GET /api/v1/ads/{id}` **عام** (بلا `auth:sanctum`)، فالـguard الافتراضي `web` يجعل `$request->user()` = null حتى مع توكن Bearer صالح → سياسة `AdPolicy::view` تحجب مسودة المالك → 404 على صفحة التعديل. (الاختبار القديم كان يستخدم `Sanctum::actingAs` الذي **يُخفي** الخلل.)
- **الإصلاح:** حلّ المستخدم صراحةً عبر guard الـ`sanctum`: `Gate::forUser($request->user('sanctum'))`. + اختبار انحدار بتوكن PAT **حقيقي** في الترويسة (يمسك الخلل فعلاً).
- **التحقق الحي:** مالك+توكن → **200**، مجهول → **404** (المسودة تبقى محجوبة عن العامة).
- **الكوميت:** `3cb34f0`

### 2.4 — لوجو الأدمن
- **الإصلاح:** حقن العلامة في التوب‑بار عبر `PanelsRenderHook::TOPBAR_START` + إخفاء علامة ترويسة الـsidebar → لوجو واحد متناسق بحجم الشريط.
- **الكوميت:** `1a533f8`, `3cb34f0`

## 4) ملاحظات وتوصيات

1. **دبلوي GitHub Actions فشل** لدفعة `3cb34f0` (تشغيل #17) — السبب: `dial tcp: i/o timeout` في خطوة "Deploy over SSH"، أي **انتهاء مهلة اتصال SSH** من runner الخاص بـGitHub إلى الـVPS (غالباً حجب مؤقت لـIP الـrunner من جدار CSF/cPHulk، أو انقطاع شبكي عابر — التشغيل #16 قبلها بـ26 دقيقة نجح). **ليس خطأ كود/إعداد.** نُشرت الدفعة يدوياً عبر `deploy/scripts/deploy-api.sh` (health 200)، ثم **أُعيد تشغيل الـworkflow ونجح** ✅ — مؤكِّداً أن الفشل عابر.
   - **تصليب مقترح:** لفّ خطوة `appleboy/ssh-action` بإعادة محاولة (مثل `nick-fields/retry`) لتجنّب فشل الدبلوي عند أي انقطاع SSH عابر مستقبلاً.
2. **مساحة القرص (بيئة التطوير) امتلأت 100%** أثناء العمل — حُرّر ~8G من كاش npm/composer. القرص لا يزال ضيّقاً (~4G)؛ يُنصح بتنظيف أعمق.
3. **عقد رفع الصور في OpenAPI** يذكر `AdEnvelope` بينما التنفيذ والاختبار يستخدمان `{ data: { images } }` — انحراف توثيقي يستحق المواءمة لاحقاً (التنفيذ مستقر ومختبَر).
4. **Playwright e2e (Part C) مؤجَّل** بقرار: تجنّب تلويث بيانات الإنتاج وضغط القرص. مهيّأ جزئياً (متصفّح Chromium منزّل + سكربت `test:e2e`)؛ يحتاج `playwright.config` + specs ضد ستاك محلي.
